
# About Pact and Contract Testing Concepts

Versioning is a normal part of software development. Each release of an application, or even each Git commit, can be
treated as a new version of that application.

In Pact, it helps to keep these three version types separate:

- Consumer version
- Pact version
- Provider version

## Versions

### Consumer Version

A consumer version is a version of the consumer application's source code. It is often identified by a Git SHA, build
number, or release version.

Example: if a web app changes some UI buttons or adds an optional form field that is not sent to the provider API, the
consumer version changes because the application code changed.

However, the pact version does not change unless the consumer's expectations of the provider API changed.

### Pact Version

A pact version is a version of the contract generated by the consumer tests for a specific provider.

The pact changes when the consumer's expectations of the provider change.

Example: if the consumer now expects the provider API to return an `additional_email` field, the consumer version changes
and the pact version changes too. The generated pact now contains a new expectation.

| Consumer version | Code change                                      | Pact with UserService |
|------------------|--------------------------------------------------|-----------------------|
| `abc111`         | Expects `GET /users/1` returns `id`, `name`      | Pact version 1        |
| `def222`         | Refactor UI button only                          | Pact version 1        |
| `ghi333`         | Consumer now also expects `additional_email`     | Pact version 2        |

Important rules:

- Multiple consumer versions can have the same pact version.
- A specific consumer version will always have exactly one pact version for each provider.

### Provider Version

A provider version is a version of the provider application's source code. Like a consumer version, it is often identified
by a Git SHA, build number, or release version.

Provider source code changes are considered in Pact, but they are tracked as provider versions, not pact versions.

Example: if the provider changes its database implementation but still returns the same API response, the provider version
changes but the pact version does not change.

| Provider version | Provider change                                      |
|------------------|------------------------------------------------------|
| `api-111`        | Returns `id`, `name`                                 |
| `api-222`        | Internal database refactor, same API response        |
| `api-333`        | Removes `name` and returns `fullName` instead        |

### Verifying the Pact

Consumer tests create and publish pacts. Provider verification checks whether a specific version of the provider
application can satisfy a specific pact version.

Example consumer versions and pact versions:

| Consumer version | Consumer change                               | Pact with UserService |
|------------------|-----------------------------------------------|-----------------------|
| `web-aaa`        | Expects `GET /users/1` returns `id`, `name`   | Pact version 1        |
| `web-bbb`        | UI-only change                                | Pact version 1        |
| `web-ccc`        | Now also expects `email`                      | Pact version 2        |

Verification results:

| Provider version | Verified against Pact 1 | Verified against Pact 2 |
|------------------|-------------------------|-------------------------|
| `api-111`        | Pass                    | Fail                    |
| `api-222`        | Pass                    | Fail                    |
| `api-333`        | Fail                    | Fail                    |

Why:

- Pact version 1 expects `id` and `name`.
- Pact version 2 expects `id`, `name`, and `email`.
- `api-111` and `api-222` satisfy Pact version 1.
- `api-222` changed provider source code, but did not change the API contract.
- `api-333` fails because it removed `name`, which the consumer still needs.

### Short Version

- Consumer code changes create new consumer versions.
- Provider code changes create new provider versions.
- Contract content changes create new pact versions.
- Consumers create pacts.
- Providers verify pacts.
- Verification results say whether a provider version satisfies a pact version.
